    Communication Flow

    1. 1. Handshake & Initialization

      The Host initiates a connection with the Server. They exchange version info and agree on protocol details, establishing a secure communication channel.

    2. 2. GetCapabilities

      The Host sends a GetCapabilities request. The Server responds with a **JSON Manifest** detailing its available Tools, Resources, and their documentation.

    3. 3. InvokeTool / StreamContent

      Based on user or LLM intent, the Host sends an InvokeTool request to execute a function or a StreamContent request to get data. The Server processes the request and sends back the result or a stream of data.

    4. 4. Termination

      The connection is gracefully closed by either the Host or Server, terminating the session.

    The JSON Manifest: The Server's "Menu"

    The response to the GetCapabilities request is a structured JSON object, often called the "manifest". This manifest is the cornerstone of discoverability, telling the Host (and by extension, the LLM) exactly what the server can do.

    
{
  "tools": [
    {
      "name": "files_search", 
      "description": "Searches for files in the workspace by filename or content.",
      "parameters": {
        "type": "object",
        "properties": {
          "query": {
            "type": "string",
            "description": "The search query, can be a filename or content snippet."
          }
        },
        "required": ["query"]
      }
    },
    {
      "name": "files_readFile",
      "description": "Reads the content of a specific file, given its path.",
      "parameters": {
        "type": "object",
        "properties": {
            "path": {
                "type": "string",
                "description": "The full path to the file."
            }
        },
        "required": ["path"]
      }
    }
  ],
  "resources": [
    {
      "name": "file",
      "description": "Represents a file in the user's workspace.",
      "uri_template": "file://{filePath}"
    }
  ]
}

    Model Context Protocol Standards

    MCP is built on a set of core concepts and a well-defined protocol that ensures interoperability and security. This section breaks down the fundamental building blocks. For the complete, official specification, visit the Model Context Protocol website.

    Core Concepts

    Tools

    Functions or capabilities exposed by a server that an AI model can invoke to perform actions (e.g., `readFile`, `sendEmail`).

    Resources

    Data objects made available by a server, identified by a unique URI. They provide context to the AI model (e.g., file content, database records).

    Context

    The collection of all available resources and tools that an AI model can use to understand a user's request and formulate a response.

    Protocol Features

    Server Features

    • Capabilities Definition: Servers must clearly define their tools and resources in the JSON manifest.
    • Streaming Support: Servers can stream content for long-running operations or large data payloads.
    • Progress Reporting: For long-running tool invocations, servers can send progress updates to the client.
    • Error Handling: Servers must provide structured error messages that both humans and AI can understand.

    Client Features

    • Dynamic Discovery: Clients discover server capabilities at runtime by calling `GetCapabilities`.
    • Tool Invocation: Clients can execute server tools with specified parameters.
    • Content Streaming: Clients can request and handle streamed resource content from servers.
    • Security Context: The client (via the Host) is responsible for managing security, permissions, and user consent.
